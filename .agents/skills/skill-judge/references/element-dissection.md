# Element Dissection — 7 步锻造协议

> 当 SKILL.md 路由到 Mode 2 时加载此文件
> 来源：code-reviewer 精装修实战（2 个社区 Skill → 17 个元素 → 9 个精华 → 2 个绝杀组合 → 134 行交钥匙）

## 核心洞察

```
整体评分的盲区：
  77 分的 Skill 被整体拒绝 → 95 分的置信度过滤框架也被丢了
  82 分的 Skill 被整体拒绝 → 93 分的工作流映射也被丢了

元素解剖解决这个问题：
  拆开 → 逐个评分 → 精华提取 → 冗余丢弃 → 组合检测 → 精确融合
```

**按人头裁 vs 按能力裁**：整体评分是"这个部门不行，全裁"。元素解剖是"这个人 95 分调走，那个人 25 分裁掉"。

## Step 1: DISSECT — 拆为原子元素

读完候选 Skill 后，拆成原子元素。每个元素是一个**独立可移植的知识单元**。

### 元素类型

| 类型 | 定义 | 示例 |
|------|------|------|
| 思维框架 | 改变"怎么想"的知识 | 置信度过滤（>80% 才报告） |
| 决策树 | 多路径选择的判断依据 | 强制 vs 可选审查触发器 |
| 流程知识 | 专家才知道的执行节奏 | 工作流-审查频率映射 |
| 操作协议 | 标准化的调度/交互方式 | SHA-based subagent 调度 |
| 决策框架 | 结果判定的标准 | 三档审批 APPROVE/WARNING/BLOCK |
| 输出格式 | 标准化的输出模板 | Review Summary 表格 |
| 分类体系 | 信息组织的维度 | 四级 Severity |
| 检查清单 | 逐项检查的条目列表 | 安全检查 8 项 |
| 领域知识 | 特定技术栈的专项知识 | React deps / Node.js N+1 |
| 反模式 | NEVER 清单 | "别攒超过 3 个 task 才审查" |
| 教学示例 | BAD/GOOD 代码对比 | SQL 注入示例 |

### 拆解输出格式

```markdown
| # | 元素名 | 类型 | 内容摘要 |
|---|--------|------|----------|
| 1 | [名称] | [类型] | [一句话描述核心内容] |
```

## Step 2: SCORE — 逐个评分

每个元素独立评分（0-100），标准：

| 分段 | 含义 | 典型特征 |
|------|------|----------|
| 90-100 | 纯专家知识 | Claude 不会自己产生，有实操价值 |
| 70-89 | 有价值的激活知识 | Claude 可能知道但不会主动想到 |
| 50-69 | 边界知识 | 有些场景有用，但非通用 |
| 25-49 | 低价值 | 大部分是 Claude 已知内容 |
| 0-24 | 冗余 | 纯 token 浪费，应删除 |

### 评分输出格式

```markdown
| # | 元素名 | 类型 | 单独评分 | 我们体系有吗？ |
|---|--------|------|----------|----------------|
| 1 | [名称] | [类型] | [分数] | [有/无/部分覆盖 + 说明] |
```

## Step 3: COMBINE — 检测组合效应

**关键洞察**：单个元素平庸，但组合可能产生涌现价值。

### 组合检测方法

1. **依赖关系**：元素 A 是元素 B 的骨架？
   - 例：Severity 分类(75分)是 Summary 表格(88分)的骨架——没有分级，表格不知道统计什么
2. **流程串联**：多个元素首尾相接形成完整链条？
   - 例：何时触发 → 什么频率 → 怎么调度 → 怎么过滤 → 怎么输出 → 怎么判定 → 怎么响应
3. **互补覆盖**：来自不同 Skill 的元素填补彼此空白？
   - 例：Top 1 的"审查怎么做" + Top 2 的"审查何时做" = 完整生命周期

### 组合评分

```
组合价值 = max(各元素单独平均分, 组合后的完整性评估)

如果组合后覆盖一个完整生命周期/工作流 → 组合价值 > 95
如果组合后只是叠加没有涌现 → 组合价值 = 单独平均分
```

### 输出格式

```markdown
### 组合 [字母]: [组合名]

元素: [#1a + #1g + #1h + #1i]
串联: [元素A] → [元素B] → [元素C] → [元素D]
单独平均分: XX
组合价值: XX
涌现效应: [有/无] — [描述]
```

## Step 4: COMPARE — 对比现有体系

**不对比就融合 = 盲目搬家**。必须逐个元素对比现有体系。

### 对比维度

对每个精华元素（>70 分）问：

| 问题 | 判定 | 行动 |
|------|------|------|
| 我们体系里有这个元素吗？ | 没有 → **新生态位** | 直接锻造入列 |
| 有，新的严格更好？ | 是 → **替换** | 新的替掉旧的 |
| 有，差不多？ | 是 → **丢弃** | Highlander 原则，不重复 |
| 有，但互补？ | 是 → **融合** | 合并为更强版本 |
| 现有的反而更好？ | 是 → **保留现有** | 标注"现有更优"，不动 |

### 对比方法

1. 读 ARMORY.md 找同生态位武器
2. 搜现有 Skills/Agents/Rules 文件
3. 逐个元素对比，不跳过

### 输出格式

```markdown
| # | 精华元素 | 现有体系 | 判定 | 行动 |
|---|----------|----------|------|------|
| 1 | [元素名] | [有/无 + 具体文件] | [新/替换/丢弃/融合/保留现有] | [具体操作] |
```

## Step 5: FORGE — 锻造方案

基于 Step 4 的对比结果，设计精确的锻造方案。

### 设计原则

1. **最小侵入**：改最少的文件，补最精确的位置
2. **不碰更优的**：现有已经更好的部分绝不动
3. **不创建冲突**：不创建同名 Agent/Skill 覆盖（会出优先级问题）
4. **架构决策**：

| 元素性质 | 放在哪里 | 理由 |
|----------|----------|------|
| 所有审查都应遵守的标准 | Rule | Rule 每次会话自动加载 |
| 特定场景触发的工作流 | Skill | Skill 按需加载 |
| 执行特定任务的角色定义 | Agent | Agent 被调度时加载 |
| 思维框架/决策树 | Skill 或 Rule | 看通用性——通用放 Rule，专项放 Skill |

### 输出格式

```markdown
## 锻造方案

### 新建文件
| 文件 | 类型 | 融入的元素 | 行数估计 |
|------|------|------------|----------|

### 修改文件
| 文件 | 修改内容 | 融入的元素 |
|------|----------|------------|

### 不动的
| 文件/元素 | 理由 |
|-----------|------|

### 丢弃的
| 元素 | 分数 | 丢弃理由 |
|------|------|----------|
```

## Step 6: INSTALL — 写入文件

按锻造方案执行写入。每个文件写完后立即确认：

```
[ ] 内容是否只包含精华元素（无冗余）
[ ] 格式是否符合 Skill/Rule/Agent 规范
[ ] frontmatter 是否完整（name + description with WHAT/WHEN/KEYWORDS）
[ ] 是否与现有文件冲突
```

## Step 7: VERIFY — 回归验证

锻造完成后验证：

```
[ ] 新文件出现在 Skills/Rules/Agents 列表中
[ ] 不与现有同名组件冲突
[ ] ARMORY.md 已更新（新武器注册）
[ ] MEMORY.md 已更新（状态变更）
[ ] 现有功能未被破坏（特别是被融合的部分）
```

### 交钥匙报告格式

```markdown
## 精装修交钥匙报告

### 施工前后对比
| 元素 | 施工前 | 施工后 | 来源 |
|------|--------|--------|------|

### 没动的（现有更优）
| 元素 | 现有体系 | 不动理由 |
|------|----------|----------|

### 丢弃的（冗余/低分）
| 元素 | 分数 | 丢弃理由 |
|------|------|----------|

### 文件清单
| 文件 | 动作 | 行数 |
|------|------|------|
```

## 实战案例：code-reviewer 精装修

以下是首次元素解剖的完整记录，作为后续操作的参考基准。

### 输入
- Top 1: code-reviewer Agent (48,610⭐, 整体 77/120 = D 级)
- Top 2: requesting-code-review Skill (13,249⭐, 整体 82/120 = D 级)
- 两个都没过 80 分门槛，传统模式会整体拒绝

### 解剖结果
- 17 个原子元素
- 评分范围：25 ~ 93
- 9 个精华（>70 分）+ 8 个丢弃（<70 分）

### 组合检测
- 组合 A（审查输出标准化）：Severity + 输出模板 + Summary + 审批标准 → 完整输出流水线
- 组合 B（审查生命周期）：触发器 + 节奏映射 + 反馈响应 → 完整审查管理
- 组合 C（A+B 全链路）：WHEN → HOW → FILTER → FORMAT → DECIDE → RESPOND

### 对比现有体系后
- 4 个现有更优（不动）：置信度评分、结构化输出、SHA 调度、CLAUDE.md 感知
- 5 个缺失需补：触发决策树、节奏映射、Summary 表格、四级 Severity、反馈响应规则

### 锻造产物
- `requesting-code-review/SKILL.md`（82 行）— 补 WHEN + 节奏 + 响应
- `rules/code-review-standards.md`（52 行）— 补 FORMAT + 标准

### 关键数字
```
2 个 D 级 Skill → 17 个元素 → 9 个精华 + 2 个绝杀组合
→ 对比后 5 个需补 → 134 行精装交付 → 零冗余
```

这就是"按人头裁"和"按能力裁"的区别。
