# 错误纪律（3-Strike + 持久化学习）

## 3-Strike 升级协议

遇到错误时严格按梯度升级，每层只试一次后必须升级：

- **Strike 1**：诊断根因 + 定向修复。只改最小范围。
- **Strike 2**：换到**不同等价类**的方法（见下表）。上一条路不通，走另一条。
- **Strike 3**：质疑前提假设 + 搜索社区方案。可能整个方向错了。
- **3 次后**：停止尝试，将问题、已试方案、失败日志升级给用户决策。

## 方法等价类（同类内切换不算"换方法"）

| 等价类 | 包含的操作 | 换方法 = 跳到另一类 |
|--------|-----------|---------------------|
| 手动编辑配置 | 改 .mcp.json / settings.json / .env / 任何配置文件 | → CLI 命令注册 |
| CLI 命令注册 | claude mcp add / npm config set / git config | → 查文档确认正确方式 |
| 查文档 | WebSearch / context7 / 读 docs/ | → 查社区 Issue/讨论 |
| 环境变量调整 | export VAR= / 改 .env / 改 .bashrc | → 改代码中的默认值 |
| 路径调整 | 换绝对/相对路径 / 换目录 | → 验证程序是否存在 |

## 错误持久化（Reflexion Pattern）

- 任务/测试失败时，将错误模式追加到 `MEMORY.md` 或当前任务状态文件。
- 格式使用 Error Tracking Table：

```markdown
| Error | Attempt | Resolution |
|-------|---------|------------|
| 具体错误信息 | 尝试了什么 | 最终解法或"未解决" |
```

- 新会话遇到相似上下文时，先匹配历史模式再动手，匹配到记录后直接跳到上次未尝试的方案。

## 2-Action 写入规则

- 每执行 2 次查看/搜索/浏览操作后，必须立即将关键发现写入文件。
- 理由：多模态信息在 compact/上下文重置时会丢失，文件不会。
- 写入目标：`MEMORY.md`、任务状态文件、或代码注释，视场景选择。

## 防重复失败

- 出错后第一个动作：检查 Error Tracking Table 是否有同类记录。
- 有记录 → 直接跳到上次未尝试的方案，不从头来。
- 无记录 → 进入 3-Strike 流程，同时建立新记录。
