# 置信度门控（执行前必检）

## Confidence Check（每次执行前花 100-200 tokens 评估）

- **>=90%**：直接执行，不废话
- **70-89%**：列出备选方案，说明权衡，让用户选
- **<70%**：停下来提问，收集缺失信息后再动手

ROI：前置检查成本 100-200 tokens，可避免 5,000-50,000 tokens 的错误方向返工。

## 灰色地带探测（执行前主动识别并确认）

- **视觉/UI**：布局方式、信息密度、交互模式、空状态处理
- **API/CLI**：响应格式、标志位默认值、错误处理策略
- **内容系统**：结构层级、语调风格、详细程度
- **搜索/检索**：搜索范围、结果深度、输出格式

发现任何灰色地带 → 先问，不猜。猜错的代价远大于多问一句。

## Read vs Write 决策矩阵

| 场景 | 动作 | 原因 |
|------|------|------|
| 刚写了文件 | 不读 | 内容还在上下文里 |
| 看了图片/PDF | 立即写下发现 | 多模态内容会从上下文丢失 |
| 开始新阶段 | 读计划文件 | 上下文可能已过时 |
| 出现错误 | 读相关源文件 | 需要当前状态来定位问题 |
| 间隔后恢复 | 读所有规划文件 | 完整恢复断点状态 |

## 执行纪律

- 不确定时问，不猜。猜测是最贵的操作。
- 每次动手前默念：我的置信度是多少？灰色地带在哪？
