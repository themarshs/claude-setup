## 二、什么迭代、什么不迭代

### 一次性任务（做完就不变了）

| 任务 | 说明 |
|------|------|
| 装某个具体 Skill | 装 agent-browser 就一次 |
| 配某个具体 MCP | 配 context7 就一次 |
| 删不合格的 Skill | 删 browser-use 就一次 |
| 设某条具体权限 | settings.json 加一条 deny |

### 需要迭代的（每圈都应该变好）

| 维度 | 第一圈水平 | 理想进化方向 |
|------|-----------|-------------|
| **搜索能力** | `npx skills find` 单关键词 | 多渠道并行（skills.sh + GitHub + context7 + agent-browser 爬） |
| **评估能力** | 手动读 SKILL.md + 人肉判断 | skill-judge 自动评分 + 安全扫描脚本化 |
| **安装验证** | 手动 symlink + 手动 ls 确认 | 自动化安装脚本 + 冒烟测试 |
| **知识地图** | CLAUDE.md 手动更新 | 每圈自动追加新 Skill 到知识地图 |
| **多 Agent 协作** | 上次一个卡住一个中断 | 任务切分更合理、通信协议更清晰 |
| **迭代计划本身** | 线性清单 | 每圈回顾 → 修正方法 → 下圈更优 |

### 元迭代（迭代过程本身的迭代）

这是最关键也是我之前完全没考虑的。每完成一圈需要问：

1. **搜的够广吗？** — 这一圈用了什么渠道搜？漏了什么？下圈加什么渠道？
2. **评的够准吗？** — 这一圈的评估标准是什么？有没有装了不该装的？有没有漏掉好东西？
3. **装的够稳吗？** — 安装过程有没有出错？验证步骤够不够？
4. **协作够顺吗？** — 多 Agent 有没有卡住？任务拆分合不合理？
5. **计划本身对吗？** — 这个四问本身够不够？需不需要加新的回顾维度？

## 三、多 Agent 的用法

### 上次的问题

| 问题 | 原因 | 教训 |
|------|------|------|
| skills-researcher 卡住 | 13 个关键词串行搜太慢 | 任务粒度要更细，每个 agent 只做 2-3 个关键词 |
| mcp-researcher 中断 | idle 后被 interrupted | 需要更好的错误恢复机制 |
| 权限弹窗太多 | 白名单太细 | 已修复为 blanket Bash allow |
| lead 等太久 | 不知道 teammate 在干啥 | 需要进度通信协议 |

### 多 Agent 可以做什么

| 场景 | 并行方式 | 预期收益 |
|------|----------|----------|
| 搜索阶段 | N 个 researcher 各搜不同领域 | 搜索速度 ×N |
| 评估阶段 | evaluator 用 skill-judge 批量打分 | 评估标准统一 |
| 安装阶段 | installer 逐个装 + 验证 | 与搜索评估解耦 |
| 回顾阶段 | reviewer 分析本圈成果写报告 | lead 专注决策 |

### 尚未解决的问题

1. **Windows 上 Agent Teams 体验差** — 没有分屏，只有 in-process 模式
2. **teammate 可靠性** — 上次有卡死的，需要超时机制
3. **成本** — 每个 teammate 都消耗 token，搜索类任务能否用 haiku？
4. **什么时候用、什么时候不用** — 简单任务不需要多 agent，判断标准是什么？

## 四、螺旋模型详解

### 每圈的结构

```
搜索 → 评估 → 安装 → 验证 → 文档更新 → 回顾
  ↑                                          |
  ←————————— 回顾反馈改进下一圈 ——————————————↙
```

### 圈与圈之间的进化

| | 第一圈 | 第二圈 | 第三圈 |
|---|--------|--------|--------|
| 搜索工具 | npx skills find | + context7 MCP 查文档 | + agent-browser 爬 GitHub |
| 评估方法 | 手动读 SKILL.md | + skill-judge 打分 | + 自动安全扫描 |
| 协作模式 | 单 agent | 2 agent 并行搜 | 分工更细（搜/评/装/审） |
| 知识积累 | 零基础 | CLAUDE.md 有基本地图 | 有完整武器库索引 |
| 迭代方法 | 线性清单 | 有回顾环节 | 回顾环节本身被优化过 |

### 每圈的搜索域

| 圈数 | 搜索目标 | 搜索渠道 |
|------|----------|----------|
| 1 | 检索类 Skills | skills.sh |
| 2 | 获取能力 Skills（browser、crawl） | skills.sh + GitHub |
| 3 | 分析能力 Skills（RAG、search） | skills.sh + GitHub + context7 |
| 4 | MCP 生态 | Anthropic MCP Registry + GitHub |
| 5 | 自造 Skill 填补空白 | 基于前4圈发现的缺口 |
| N | 持续演进 | 用已有能力发现新能力 |

## 七、当前计划的不足（自我批评）

1. **不是螺旋** — 只是一个线性安装清单，圈与圈之间没有反馈回路
2. **没有元迭代** — 没有"回顾迭代方法本身"的环节
3. **多 Agent 是摆设** — 上次用了但没用好，这次的计划里甚至没提
4. **搜索渠道单一** — 始终依赖 `npx skills find`，没有用已装工具去扩展搜索
5. **评估没标准化** — 有 skill-judge 但没在流程里强制使用
6. **知识没累积** — 每圈的发现没有系统性地变成下一圈的输入
7. **没有终止判断** — 不知道什么时候该从"建武器库"转向"用武器库"
