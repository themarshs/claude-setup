## 一、大模型的根本性缺陷（共识）

| 缺陷 | 表现 | 可否由用户弥补 |
|------|------|---------------|
| 失忆 | 每次会话上下文归零 | **可以** — CLAUDE.md / Rules |
| 无行动力 | 本质是文本进文本出，需外挂工具 | **可以** — Tools / MCP |
| 无实时信息 | 训练数据有截止日期 | **可以** — WebSearch / MCP 接实时源 |
| 行为不稳定 | 同样指令不同次执行结果不一致 | **可以** — Skills 固化流程 |
| 无风险意识 | 不会主动判断操作危险性 | **可以** — Hooks 做外挂刹车 |
| 上下文有限 | ~200k tokens 窗口，大项目放不下 | **可以** — 按需加载，分层注入 |
| 推理能力上限 | 模型架构和训练决定的天花板 | **部分可以** — 见第三节 |

## 二、用户可控 vs 不可控（共识）

```
用户可控（上下文基础设施）          用户不可控（厂商侧）
─────────────────────            ──────────────
记忆    → CLAUDE.md / Rules       推理能力上限（模型权重）
检索    → MCP / Skills / 文档      模型版本迭代
流程    → Skills / Hooks           架构升级
知识    → Rules / 上下文注入
约束    → Hooks / 治理规则
```

核心结论：用户的投入应集中在左列——建设"上下文基础设施"，让每次会话都能在正确的知识和约束下发挥最大推理能力。右列的改进由厂商推动，且左列建设完成后能自动受益于右列升级。

类比：不是"训练模型"（改权重），而是"编写一套持久化的提示词体系"（改输入）。

## 三、方法论能否提升推理能力？（已达成共识）

### 用户观点

"三个臭皮匠顶个诸葛亮"——好的架构可以放大和叠加个体能力。即使推理能力本身有天花板，通过方法论规范化（如：拆解问题 → 模块化求解 → 分步验证），可以让实际输出质量逼近甚至突破单次推理的上限。

数学题类比：普通学生靠直觉试错，高手把问题拆成已知模块后规范化求解——同样的知识储备，方法论不同，结果天差地别。

### Claude 回应

**同意的部分：** 方法论确实能提升实际输出质量。AI 领域的 Chain-of-Thought、Tree-of-Thought 已经实证了这一点。在 CLAUDE.md/Skills 中写入结构化推理框架，能让模型更稳定地接近能力上限。

**修正的部分：** "三个臭皮匠"效应成立的前提是异构——不同的信息和视角。同一个模型模拟三个角色，盲区是重叠的。真正的增益来自 **用户 + 模型 + 外部约束系统（Hooks/Rules）** 构成的异构团队。

### 达成的共识

```
方法论注入（CLAUDE.md/Rules/Skills）
    → 提升：稳定性，让模型更一致地接近能力上限
    → 提升：结构化程度，减少遗漏
    → 不提升：能力上限本身

架构设计（用户 + 模型 + Hooks + MCP）
    → 提升：系统整体输出，超过任何单一节点的上限
    → 这才是真正的"三个臭皮匠"效应
```

## 四、方法论缺失的实证（第 5 轮对话）

### 用户批评

用户指出 Claude 在讨论"从哪里下手写 CLAUDE.md"时，直接凭已有认知输出方案（"分三块写"），没有先检索 `D:/ai/setup/docs/` 下的官方文档。这恰恰是方法论缺失的表现。

### Claude 自检

确认这是一个典型错误：

```
错误路径：接到任务 → 凭认知直接输出方案 → 遗漏关键信息
正确路径：接到任务 → 检索现有最佳实践 → 消化吸收 → 再输出方案
```

### 检索官方文档后发现的修正

阅读 `best-practices.md`、`memory.md`、`skills.md` 后，发现多处之前会说错或遗漏的要点：

| 之前的认知（有偏差） | 官方文档的实际建议 |
|---|---|
| CLAUDE.md 分三大块全塞进去 | **要精简**，每行问"删掉会犯错吗？"太长会导致指令被忽略 |
| 写一个全局 CLAUDE.md 就行 | 记忆系统是**分层**的：全局/项目/私有/Rules/自动记忆 |
| 方法论全放 CLAUDE.md | 高频通用的放 CLAUDE.md，特定流程放 **Skills 按需加载** |
| 从零开始写 CLAUDE.md | 官方有 **`/init` 命令**可以自动生成初版再精炼 |
| Skills 是方法论的载体 | Skills 是**按需加载的知识和流程**，不等于方法论 |

### 核心教训

**"先检索，再输出"应该成为写进 CLAUDE.md 的第一条方法论规则。**

这条规则本身就是用户所说的"方法论可以提升实际输出质量"的直接证据——同样的推理能力，加了这一步检索，结果完全不同。

## 五、记忆系统分层架构（来自官方文档）

| 层级 | 位置 | 用途 | 加载时机 |
|------|------|------|---------|
| 组织策略 | 系统级 CLAUDE.md | 全组织强制规则 | 每次会话 |
| 全局个人 | `~/.claude/CLAUDE.md` | 所有项目通用偏好 | 每次会话 |
| 全局规则 | `~/.claude/rules/*.md` | 个人通用规则 | 每次会话 |
| 项目共享 | `./CLAUDE.md` | 团队协作规范 | 每次会话 |
| 项目规则 | `.claude/rules/*.md` | 模块化规则（支持 glob 匹配） | 按文件匹配 |
| 项目私有 | `./CLAUDE.local.md` | 个人项目偏好 | 每次会话 |
| 自动记忆 | `~/.claude/projects/<project>/memory/` | Claude 自己写的笔记 | 前 200 行 |

**关键设计原则：**
- CLAUDE.md 每次都加载 → 必须精简，只放"删掉会犯错"的内容
- Skills 按需加载 → 放特定流程和领域知识
- Rules 按 glob 匹配 → 放文件类型相关的规范
- 信息越具体，放的层级越深

## 六、检索能力的两个维度（第 6-7 轮对话）

### 用户提出的核心问题

检索问题的关键在于两点：**什么时候触发检索** 和 **检索能力的武装**。

### 触发时机（什么时候开枪）

模型最大的盲区是"不知道自己不知道"，会直接凭已有认知输出，看起来自信但可能错误。

触发检索的场景：

```
1. 接到新任务时          — 有没有现成方案？有没有最佳实践？
2. 要动手改东西之前      — 现状是什么？依赖关系是什么？
3. 遇到不确定的判断时    — 我的认知是不是过时了？
4. 输出方案之前          — 有没有人做过类似的？效果怎样？
5. 犯错被纠正之后        — 为什么错了？是信息缺失还是推理错误？
```

这些触发条件本身就是方法论，必须写进 CLAUDE.md 最醒目的位置。

### 武装程度（打不打得准）

| 武装层级 | 内容 | 承载形式 |
|---------|------|---------|
| 内建工具 | Grep/Glob/Read/WebSearch | 已有，不需要配 |
| 知识地图 | 信息在哪、文档结构、去哪找什么 | CLAUDE.md |
| 领域知识 | 特定场景的检索策略和路径 | Skills（按需加载） |
| 外部连接 | 连通外部系统获取实时数据 | MCP |

### 两个维度的关系

```
触发时机（方法论）决定的是 → 开不开枪
武装程度（能力建设）决定的是 → 打不打得准

两者缺一不可：
- 有触发无武装 → 知道该查，但查不到、查不准
- 有武装无触发 → 有能力查，但想不到去查
```

### 检索的三层本质（不是选择题）

| 层次 | 内容 | 承载形式 |
|------|------|---------|
| 意识层 | "我应该先查一下" | 方法论 → CLAUDE.md |
| 导航层 | "这个信息在哪里" | 知识地图 → CLAUDE.md + Rules |
| 能力层 | "我能连上那个系统" | 工具 → MCP + Skills + 内建工具 |

官方文档原话佐证：
> **MCP** gives Claude the ability to interact with external systems.
> **Skills** give Claude knowledge about how to use those tools effectively.

MCP 提供连接能力，Skills 教怎么用。两者配合，不是替代关系。

## 七、软约束 vs 硬约束（第 8 轮对话）

### 用户提出的核心问题

CLAUDE.md 是软约束，要区分什么是软约束、什么是硬约束，各自适用场景和缺陷是什么。

### 定义对比

| | 软约束 | 硬约束 |
|---|---|---|
| **载体** | CLAUDE.md / Rules / Skills | Hooks (exit code 2) / Permissions |
| **执行机制** | 写进上下文，靠模型"自觉"遵守 | 外部脚本拦截，模型根本执行不了 |
| **可靠性** | 大部分时候遵守，但可能遗忘或违反 | 100%，不存在"忘了"的可能 |
| **灵活性** | 高，能理解上下文做判断 | 低，只能做规则匹配 |
| **上下文代价** | 占 tokens | 零（Hooks 在外部运行） |

### 软约束的失效场景

1. **上下文太长，指令被"淹没"** — CLAUDE.md 超过 500 行，关键规则被忽略
2. **指令和任务冲突时** — 用户说"帮我配置环境变量"，模型可能觉得改 .env 合理
3. **compact 之后** — 上下文压缩时，软约束可能被总结掉
4. **模型推理错误** — 理解了规则但判断错了适用场景

本质：**软约束是建议，不是法律。** 依赖模型的理解力和注意力，两者都不 100% 可靠。

### 硬约束的缺陷

1. **没有判断力** — 正则匹配不理解意图，文件名叫 `fix-env-bug.py` 也会被拦
2. **维护成本** — 每条规则都要写脚本，规则多了脚本难管
3. **不能处理模糊场景** — "大规模重构前要先确认"写不成硬约束
4. **权限疲劳** — 拦截太多会让人烦，变成无脑点确认

本质：**硬约束是围墙，不是大脑。** 能拦住明确危险，但不能做需要判断的决策。

### 选择标准

```
问自己一个问题：如果这条规则被违反一次，后果是什么？

后果不可逆 / 灾难性的  → 硬约束（Hooks）
  例：删除生产数据、push --force 到 main、泄露 API Key

后果可逆 / 只是不规范  → 软约束（CLAUDE.md）
  例：代码风格不一致、commit message 格式不对、没有先检索就动手

需要判断力的          → 软约束 + 人工确认
  例：大规模重构、架构变更、涉及多个系统的操作
```

### 与 OpenCode IronGate Visa 分级的对应

| 变更级别 | OpenCode 做法 | Claude Code 对应 |
|---------|-------------|-----------------|
| S（小） | 自动放行 | 软约束（CLAUDE.md 规范） |
| M（中） | 需要 Visa | 软约束 + Hooks 提示确认 |
| L（大） | 层层审批 | 硬约束（Hook 阻断 + 等用户确认） |

### 中间地带：智能约束（prompt/agent Hook）

官方文档提供了介于软硬之间的机制：

```
硬约束（command Hook）  ：正则匹配，无脑拦截，100% 可靠但无判断力
智能约束（prompt Hook） ：用轻量模型（Haiku）判断该不该拦，独立于主对话不受上下文污染
智能约束（agent Hook）  ：用子 agent 检查文件/运行命令后再判断，能力更强但更慢
软约束（CLAUDE.md）     ：靠主模型自觉，灵活但不可靠
```

### 四层防护架构（最终共识）

```
层次          载体                适用场景                     特点
─────         ────                ────────                     ────
硬约束        command Hook        不可逆操作                   100% 可靠，无判断力
                                  （删文件/force push/碰 .env）

智能约束      prompt/agent Hook   需要判断的中风险操作          独立判断，不受主上下文污染
                                  （大规模修改/跨系统操作）

软约束        CLAUDE.md/Rules     方法论/规范/偏好              灵活有判断力，但可能遗忘
                                  （代码风格/检索习惯/协作偏好）

按需知识      Skills              特定流程/领域知识              不占常驻上下文，按需加载
                                  （部署流程/特定项目检索策略）
```

各层有各自的缺陷，但叠加形成完整防线——异构的多层防护，回应了"三个臭皮匠"的核心思想。

## 九、解决方案的定调（第 9 轮对话）

### 用户批评

之前讨论的"怎么做"部分全是 Claude 基于已有认知推出来的——又犯了"先检索再输出"的老问题。问题分析可以靠推理碰撞，但解决方案应该先检索、先看别人怎么做的。

### 用户的方法论方向：Agent Teams

用户看到官方的 Agent Teams 方案，提出了核心思路：

**由一群 AI 协作办公，每一步的输出成为下一步的输入，逐级递增完成任务。**

这与之前讨论的"三个臭皮匠"思想一脉相承——不是一个模型独自推理，而是多个独立 agent 各自工作、互相挑战、汇总结论。

Agent Teams 官方架构：
```
Team Lead（主 agent）
    ├── Teammate A（独立上下文，独立工作）
    ├── Teammate B（独立上下文，独立工作）
    └── Teammate C（独立上下文，独立工作）

协调机制：
- 共享任务列表（task list）
- 点对点通信（mailbox）
- 自主认领任务（self-claim）
- Lead 汇总结论
```

关键优势：
- 每个 teammate 有独立上下文，不互相污染
- 天然异构——不同 teammate 从不同角度看同一个问题
- 支持 Hooks 做质量关卡（TeammateIdle / TaskCompleted）

当前状态：**实验性功能**，需设置 `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` 启用。

### 定调

**解决方案不靠 Claude 自己推理，要先检索现有最佳实践和官方方案，再结合实际情况落地。**

Agent Teams 思路作为后续方法论架构的参考方向，暂不立即实施，先从最简单的事情开始。

### 第一个落地动作：findskills

在讨论架构和方法论之前，先装一个最简单的 Skill 跑通流程——验证 Skills 机制本身能正常工作，同时为后续武装检索能力做准备。

## 十、方法论缺失的第三次实证（第 10 轮对话）

### 事件

Claude 调研 Skills 生态时，刚装完 `find-skills` 却没用它，而是手动用 WebFetch 抓网页。用户指出后 Claude 自检：这是同一个错误的第三次发生。

第一次：要写 CLAUDE.md，没先看官方文档
第二次：要调研 skills.sh，没先看手上有什么现成工具
第三次：刚装完 find-skills，调研 skills 的时候没用它

### 方法论规则的精化

```
不只是"先检索再输出"

```

三次错误本质相同：**有武装但没触发**。工具在手边，想不到用。

## 十一、完整架构模型：Agent Teams + Skills + MCP（第 11-12 轮对话）

### 用户提出的完整链路

```
用户 Prompt
    ↓
Lead Agent（拆分 + 调度 + 装备规划）
    ├── 分析任务性质
    ├── 拆成子任务
    ├── 决定每个子任务需要什么 Skills / MCP / 工具
    └── 分配给 Teammates（最小化装备）
         ↓
Teammate A ← Skills: systematic-debugging + MCP: Sentry
Teammate B ← Skills: test-driven-development + MCP: GitHub
Teammate C ← Skills: verification-before-completion
         ↓
各自执行，互相通信，结果汇总
         ↓
Lead 综合输出
```

每个 Teammate 是一个**被特定 Skills 武装过的独立 Agent**，Skills 告诉它怎么干，MCP 给它工具去干。

### 概念汇总表

| 讨论过的概念 | 在架构中对应什么 |
|---|---|
| 三个臭皮匠 | Agent Teams，异构 Teammates |
| 方法论注入 | 每个 Teammate 加载对应的 Skills |
| 检索能力武装 | MCP + 内建工具 |
| 记忆 | CLAUDE.md，所有 Teammate 共享 |
| 软约束 | CLAUDE.md / Rules |
| 硬约束 | Hooks，对所有 Teammate 生效 |
| 逐级递增 | 任务依赖链，上一步输出是下一步输入 |

### Token 成本约束（关键）

用户指出：如果每个 Teammate 全量加载上下文，开 5 个就是 5 倍成本，直接爆掉。

**省 token 不是优化，是架构约束。** Lead 的规划职责不只是拆任务，还要**规划每个 Teammate 的最小装备清单**：

```
Lead 的规划输出格式：

Teammate A：
  - 任务：查 bug 根因
  - Skills：systematic-debugging（只这一个）
  - MCP：Sentry（只连这一个）
  - 上下文：只给相关文件路径，不给全项目

Teammate B：
  - 任务：写修复代码的测试
  - Skills：test-driven-development
  - MCP：无
  - 上下文：只给 bug 相关模块
```

### Token 约束反向解释了所有之前的设计原则

| 设计原则 | 驱动力 |
|---------|--------|
| CLAUDE.md 必须精简 | 所有 Teammate 都加载，长了就是 N 倍浪费 |
| Skills 按需加载 | 每个 Teammate 只装它需要的 |
| Rules 按 glob 匹配 | 只有碰到对应文件的 Teammate 才触发 |
| MCP 分 scope | 不是每个 Teammate 都需要连所有外部系统 |
| 知识分层而非全量 | 全量 × N 个 Teammate = 灾难 |

**结论：精简、分层、按需——这三个原则的根本驱动力是 token 成本约束。**
